# Security Policy

## Supported Versions

The latest stable release of FileKit receives security updates and bug fixes.

Older releases are considered unsupported unless explicitly stated.

| Version | Supported |
|---------|-----------|
| Latest Release | ✅ |
| Older Releases | ❌ |

## Reporting a Vulnerability

If you discover a security vulnerability in FileKit, please do **not** create a public GitHub issue.

Instead, report it privately by emailing:

**YOUR_EMAIL_HERE**

Please include:

- Description of the vulnerability
- Steps to reproduce
- Impact assessment
- Proof of concept (if available)
- Suggested mitigation (optional)

You will receive an acknowledgment as soon as possible.

## Disclosure Policy

After the issue has been investigated and resolved, the vulnerability may be publicly disclosed with appropriate credit to the reporter, unless anonymity is requested.

## Scope

Examples of security issues include:

- Remote code execution
- Privilege escalation
- Command injection
- Path traversal
- Arbitrary file access
- Insecure update mechanisms
- Authentication or authorization bypass
- Dependency-related vulnerabilities

## Third-Party Components

FileKit depends on several open-source libraries and runtimes, including Electron.js, React, Java, and other community-maintained packages.

Security issues affecting these dependencies should also be reported to their respective maintainers when appropriate.

## Responsible Disclosure

Please allow a reasonable amount of time to investigate and resolve reported vulnerabilities before making them public.

Thank you for helping keep FileKit secure.
